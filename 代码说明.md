## 三、逐个文件讲解

### 1. `config.py` —— 全局常量与 schema

所有模块共享的常量集中在这里，不散落到各处。

| 常量 | 含义 |
|------|------|
| `EMOTION_LABELS` | 情绪标签列表（happy/sad/angry/anxious/...） |
| `FATIGUE_LEVELS` | 压力/疲劳等级（low/medium/high） |
| `FUSION_SCHEMA_EXAMPLE` | 融合层 ④ 输出的标准 JSON 格式示例 |
| `AUTO_INTERVAL_SEC` | 自动模式捕捉间隔，当前 `5`（秒） |
| `SOOTHE_EMOTIONS` | 被「舒缓类」音乐对待的情绪集合（mock LLM 用） |
| `MOCK_EMOTION` | 真实模块未就绪时 mock 感知固定返回的情绪（`neutral`，不随机） |
| `MOCK_FATIGUE` | 真实模块未就绪时 mock 感知固定返回的压力等级（`low`） |

> 改自动捕捉间隔、增删情绪标签，都改这里，**不要在别的文件里另写一份**。

### 2. `mocks.py` —— 6 个模块的占位实现

每个真实模块还没写好时，`app.py` 自动用这里对应的 `MockXxx` 类顶上。
mock 类的**函数名、参数、返回字段都和真实模块的接口契约完全一致**，所以 `app.py` 调用时不用关心拿到的是真的还是 mock。

- `MockFace.predict / gradcam`、`MockFatigue.predict`、`MockSpeech.predict`：**不随机**，固定返回 `config.MOCK_EMOTION`(neutral) / `MOCK_FATIGUE`(low) 这个舒缓/中性状态
- `MockFusion.fuse`：**真逻辑不是随机**——置信度加权投票选主导情绪，疲劳透传（可直接作为融合层参考实现）；感知 confidence 为 0 时回退到固定舒缓状态
- `MockLLM.infer`：按情绪给一段 mock CoT 推理 + music spec
- `MockMusic.generate`：**直接返回空白（`None`），不乱生成占位音符**——真实音乐模块没写好时 UI 音频栏为空，不播放任何声音


### 3. `app.py` —— 集成层（本项目核心）

#### (a) 动态加载：真实模块 or mock 自动切换
`_load()` 尝试 `import` 成员文件，成功用真实模块，失败（文件不存在/依赖没装/报错）回退到 `mocks.py` 对应类。

#### (b) pipeline 数据流
```
画面帧 ──┬──► ① 人脸情绪 (FACE.predict)   ──┐
        │                                   │
        └──► ② 疲劳/压力 (FATIGUE.predict) ──┤
                                            ├──► ④ 融合 (FUSION.fuse) ──► 统一状态 JSON
录音   ────► ③ 语音情绪 (SPEECH.predict) ──┘            │
                                                        ▼
                                          ⑤ LLM 推理 (LLM.infer)
                                                        │
                                                        ▼
                                          ⑥ 音乐生成 (MUSIC.generate) ──► 🎵
⑦ GradCAM (FACE.gradcam) 旁挂在人脸模块上，输出面部热力图
```
拆成两个辅助函数：
- `_perceive_and_fuse(image, audio)` → 跑 ①②③⑦ + ④，返回 `(perception, heatmap, state)`
- `_reason_and_compose(state)` → 跑 ⑤⑥，返回 `(reasoning_text, music)`

#### (c) 两种工作模式
- **自动模式 Auto（默认）**：摄像头/麦克风持续运行，**摄像头流式输入直接驱动** `auto_step`（`auto_cam.stream(..., stream_every=AUTO_INTERVAL_SEC)`，当前每 5 秒处理一帧）跑完整 pipeline，音乐 `autoplay`，陪伴一直存在直到关网页。每次处理都会在 `live status` 框刷新一个时间戳「心跳」，方便一眼看出在跑。
- **手动模式 Manual**：保留拍照/上传 + 录音/上传窗口，点 **▶ Run Pipeline once** 跑一次。切到手动时摄像头区隐藏、流式停止。

顶部 **Mode** 单选框切换，两模式共用同一套输出区。

#### (d) 关键设计：情绪/压力不变就不换音乐
```python
def _state_key(state):
    return (state.get("dominant_emotion"), state.get("fatigue"))
```
`auto_step` 比较本轮 `key` 与上一轮 `last_key`（存在 `gr.State`）：相同 → 音乐输出返回 `gr.skip()`，当前曲子继续播不打断；不同 → 重新推理 + 生成新音乐。

#### (e) 自动模式的两个易错点（已修复，组员勿改回）
#### (e) 自动模式的两个易错点（已踩过坑，组员勿改回）
- **摄像头流式直接驱动 pipeline，不要用 `gr.Timer` + State 中转**：早期版本让 `gr.Timer` 去读流式摄像头组件的值，结果回调里读到的是 `None`，`auto_step` 一直走"摄像头没就绪→全 skip"分支，**输出永远不刷新**。现改为 `auto_cam.stream(fn=auto_step, ..., stream_every=AUTO_INTERVAL_SEC)`，每帧直接作为参数送进来，不会是 None。
- **流式麦克风必须 `type="numpy"`**：流式音频不能用 `type="filepath"`（会每秒 422 报错刷屏）。`accumulate_audio` 把 numpy chunk 累积进缓冲，`buffer_to_wav` 在处理时合并写成临时 wav 再交给语音模块。
- **mock 状态固定时输出"看似冻结"是正常的**：mock 感知固定返回 `neutral`/`low`，所以第一次之后每次 `key == last_key`，走"不变不换"分支。`live status` 框的时间戳心跳会持续跳动，证明 pipeline 在跑。

## 四、怎么确认自动模式跑通了？（别人都还没写的情况）

自动模式不依赖别人——6 个模块缺失时全走 mock，现在就能独立验证。

**第 0 步**：确认 gradio 版本
```bash
python -c "import gradio as gr; print(gr.__version__, hasattr(gr,'Timer'), hasattr(gr,'skip'))"
```
版本 ≥ 4.39 且两个 `True`。

**第 1 步**：`python app.py`，控制台应有 6 行 `[mock]` 且不报错。

**第 2 步**：浏览器打开页面（默认 Auto），**允许摄像头和麦克风权限**，什么都不点盯着看：
- ✅ `live status` 框里的**时间戳每 `AUTO_INTERVAL_SEC` 秒（当前 5）跳动一次**——这是判断 pipeline 在跑的最直接信号
- ✅ `Perception output` / `unified state JSON` 跟着刷新（mock 固定 neutral/low，值不变但时间戳在动）
- ⚠️ **音乐栏会是空白**——预期：音乐 mock 不乱生成占位音符，等真实 `music_gen.py` 写好才有声音
- ⚠️ 状态固定为舒缓/中性（mock 不随机），第一次之后一直 `state unchanged`，这是正常的
- ✅ GradCAM 栏出现摄像头画面

> 💡 改捕捉间隔在 `config.py` 的 `AUTO_INTERVAL_SEC`（当前 = 5 秒）。
> 💡 看不到时间戳跳动 = pipeline 没在跑，多半是没给摄像头权限，或 gradio 版本太旧。

**第 3 步**（不需要 gradio，命令行验证「不变不换」）：
```bash
python -c "from app import auto_step; import numpy as np; img=np.zeros((8,8,3),dtype='uint8'); k=('neutral','low'); print('相同key→应跳过:', auto_step(img,[],k)[4]); print('不同key→应重算:', auto_step(img,[],('x','y'))[4])"
```
相同 key → 第 5 个返回值是 `skip`（音乐不动）；不同 key → 走重算分支，音乐为 `None`（mock 留空白，真实模块写好后才出声）。

## 六、常见问题

**Q：输出端一直刷 `StarletteDeprecationWarning: HTTP_422_UNPROCESSABLE_ENTITY ...`，而且看不到 output？**
旧版本里流式麦克风用了 `type="filepath"` 导致每秒 422。现已改为 `type="numpy"` + State 中转摄像头帧，更新后重跑即可。若仍有零星该 warning 但功能正常，那只是 gradio 内部的弃用提示，无害。

**Q：报错 `gradio has no attribute 'Timer' / 'skip'`？**
版本太旧，`pip install -U "gradio>=4.39"`。

**Q：我还没写完模块，会影响别人吗？**
不会，自动走 mock。

**Q：怎么确认用上了我的真实实现？**
看启动控制台那行是 `[OK ]` 还是 `[mock]`。

**Q：自动模式音乐为什么没换？**
本轮「情绪+压力」和上轮一样——预期行为（不打断当前音乐）。另外 mock 感知固定返回舒缓/中性状态（不随机），所以一直不会变。

**Q：音乐栏一直是空白 / 没声音？**
真实 `music_gen.py` 还没写好时，音乐 mock 故意返回空白（`None`），不乱生成占位音符。等音乐模块写好就有声音了。

**Q：改自动捕捉间隔？**
改 `config.py` 的 `AUTO_INTERVAL_SEC`。

**Q：端口被占 / 要公网链接？**
`app.launch(server_port=7861)` 换端口；`app.launch(share=True)` 生成临时公网链接。
