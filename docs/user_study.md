# EmotiCompanion User Study 方案

**多模态情绪感知的实时音乐陪伴系统 · 用户研究 · AIAA 3800 · HKUST(GZ) · 2026**

> 感知层（人脸/语音/融合）已用定量消融评估（[experiment_plan.md](experiment_plan.md)）。本 user study 评估的是**整套系统用起来的真实体验**——用户对着摄像头说一句话 → 系统感知情绪+疲劳 → LLM 推理需求 → 生成并播放音乐 → 用户反馈。目的是拿到"感知准不准、音乐合不合心情、推理值不值、系统有没有用、隐私能不能接受"这些**只能问人**的答案。
> 本文档按五点组织：**§1 假设 · §2 完整流程 · §3 记录哪些数据 · §4 问卷 · §5 结果分析**。**核心原则：让用户用完整系统，再问反馈——不做实验室式的盲评 A/B。**

---

## 0. 设计速览

| 项 | 内容 |
|----|------|
| **形式** | 用户**实际使用完整系统一次**（**手动模式**：一次输入→一次生成→稳定播放，见 §2），再填一份简短问卷 + 简短访谈 |
| **样本量** | **8–10 人**（每位组员各招募 ~2–3 人）。便利抽样 |
| **情绪控制** | **每人只做一种**：默认 **B 情绪诱导**（提前分配轮换目标情绪、保覆盖），强真实情绪者用 **A 自然使用**。样本层面 A+B 结合，见 §6 |
| **量表** | 5 点 Likert（1=完全不 … 5=非常），与组内既定一致 |
| **数据留存** | 默认只存**情绪标签/文本 + 打分**（**不存**原始摄像头帧/录音）。若开启 `STUDY_SAVE_MEDIA=1`（**需被试同意**），额外留存 **GradCAM 热力图⑦ + 疲劳关键点标注图②**（人脸衍生图）作分析素材——见附录 A |
| **语音后端** | `app.py` **默认已用 emotion2vec+**（跨数据集表现最好），无需手动设置 |
| **时长** | 每人 ≈ 12–15 分钟 |
| **产出** | 一个 `user_study.csv`（每人一行）+ 开放文字反馈 |

## 1. 假设

每条假设绑定一个问卷题（Q1–Q5 沿用最初规划），判定以**样本趋势**为主（N 小，不追显著）。

| 编号 | 假设 | 对应题 | 评估的模块 | 确认标准（N≤10 务实判定） |
|------|------|--------|-----------|---------------------------|
| **H1** | 系统感知的情绪与用户当时**实际感受一致** | Q1 | ①③④ 感知+融合 | Q1 中位数 ≥4/5，且 ≥60% 打 4–5 |
| **H2** | 生成的音乐**符合用户当时心情** | Q2 | ⑥ 生成 | Q2 中位数 ≥4/5 |
| **H3** | **ToM+CoT 推理**比"情绪→曲风"直接查表**更贴心** | Q3 | ⑤ 推理 | ≥60% 的人选 ToM 版更贴心 |
| **H4** | 用完一轮后用户**情绪状态有正向变化**（放松/被安抚） | Q4 | 完整系统 | Q4 中位数 ≥3.5，正向为主 |
| **H5** | 摄像头/麦克风感知的**侵扰感可接受** | Q5 | ⑧ 交互/隐私 | 描述统计；作 critical reflection 素材 |
| **H6**（可选） | **疲劳感知**让推荐更贴合用户精力状态 | Qf | ② 疲劳 | Qf 中位数 ≥3.5（含疲惫者子集） |

> H3 是原规划里唯一的"对比"，保留但**做轻量**（§2 步骤 4）：不搞盲评配对矩阵，只在用户用完后，额外让 TA 听一段"直接查表"版为**同一状态**生成的音乐，二选一。
> H6 是疲劳②的价值验证——experiment_plan §6 说过，"加疲劳更好"唯一正确的证据是**用户偏好/主观恰当度**，不是疲劳检测准确率。

## 2. 完整实验流程（每人 ≈ 12–15 分钟）

| 步骤 | 时长 | 主试做什么 | 被试做什么 | 记录 |
|------|------|-----------|-----------|------|
| **0 知情同意 + 基本信息** | 2′ | 念要点：会临时开摄像头/麦克风做感知，但**不保存**图像录音；可随时退出。收基本信息 | 阅读、签字、填年龄段/性别/听力 | consent=Y；`age_band`/`gender`/`hearing_ok` |
| **1 情绪进入**（每人一种，见 §6） | 1–2′ | 默认按分配播一段 60–90s 目标情绪短片（方案 B）；若被试已带强烈真实情绪则跳过、用方案 A | 观看进入 / 直接带真实状态 | `induction` 列 |
| **2 使用系统** | 3–4′ | 打开 `app.py`、**切到手动模式 Manual**（一次输入→一次生成→音乐稳定播放，**不会像自动模式那样中途被切换**，更适合评分）。语音后端默认 emotion2vec；**「会话标签」填参与者编号（如 P03）**；如需留存可视化则启动时加 `STUDY_SAVE_MEDIA=1`。请被试**拍一张 + 录一句当下真实的话**（如"我盯论文六小时了，脑子一片空白"），主试点 **▶ Run Pipeline once** | 提供图/音、听完整生成的音乐 | 系统输出**自动写入** `results/session_log.jsonl`；可视化（若开启）存 `results/session_media/<label>/`，见 §3 |
| **3 感知/音乐反馈** | 2′ | 问 Q1（感知准不准）、Q2（音乐合不合心情）、Q4（用完心情有无变化） | 打分 | Q1/Q2/Q4 |
| **4 推理价值对比（H3）** | 2′ | 说"再给你听一段用**简单查表**方式为你刚才状态生成的音乐"，播放 standard 版，问 Q3 更贴心的是哪段 + 为什么 | 二选一 + 说原因 | Q3 + reason |
| **5 疲劳（可选 H6）+ 隐私** | 2′ | 问 Qf（音乐是否贴合你此刻精力/疲惫程度）、Q5（被摄像头麦克风感知的侵扰感） | 打分 | Qf/Q5 |
| **6 简短访谈** | 2–3′ | 问最喜欢的一点 / 最想改进 / 对隐私的想法 | 口头回答 | 开放文字 |

**步骤 4 怎么操作**：把 UI 里的「⑤ Reasoning mode」从 **ToM+CoT** 切到 **Standard (baseline)**，**保持同一张图/同一段录音**，再点一次 ▶ Run Pipeline once——即得同一状态下的"直接查表"版音乐。步骤 2 用默认 ToM+CoT，故两段是**同输入、仅推理方式不同**的干净对比。**直接在 UI 切换即可**。

**减小顺序偏差**：被试若总是先听 ToM（步骤 2）、后听 Standard（步骤 4）会有顺序效应。可对约一半被试**在步骤 2 先用 Standard、步骤 4 再用 ToM**（开跑前把 toggle 切一下），抵消先后。

> 主试全程用统一脚本念指导语（附录 B），不评价、不暗示哪个更好。

## 3. 需要记录的数据

**一个文件 `user_study.csv`，每位被试一行**（简单、够用）：

| 字段 | 取值 | 说明 |
|------|------|------|
| `pid` | P01… | 匿名编号 |
| `date` | YYYY-MM-DD | |
| `age_band` | 18–22 / 23–26 / 27+ | 描述样本用（粗粒度、不做子组分析） |
| `gender` | 男 / 女 / 其他 / 不愿透露 | 描述样本用（可选、非必填） |
| `hearing_ok` | Y / N | 听力是否正常（判断音乐相关，可选） |
| `induction` | natural / induced:<emotion> | 每人**只做一种**（见 §6）：自然使用，或诱导的目标情绪 |
| `sys_emotion` | 系统感知的情绪 | 从界面读，用于对照 Q1 |
| `sys_fatigue` | low/medium/high | 系统判的疲劳 |
| `sys_need` | 文本 | ⑤ 推理出的需求 |
| `sys_music_spec` | 文本 | ⑥ 的音乐描述词 |
| `Q1_perc_match` | 1–5 | 感知情绪与实际一致度 |
| `Q2_music_fit` | 1–5 | 音乐符合心情度 |
| `Q3_pref` | tom / std / tie | 哪个版本更贴心 |
| `Q3_reason` | 文本 | 为什么 |
| `Q4_mood_change` | 1–5 | 用完情绪正向变化（1=更糟…3=没变…5=明显变好） |
| `Q5_privacy` | 1–5 | 侵扰感（1=很强…5=完全不介意） |
| `Qf_fatigue_fit` | 1–5（可选） | 音乐贴合精力/疲惫程度 |
| `like_text` / `improve_text` | 文本 | 访谈开放反馈 |

> **`sys_*` 从哪来、放哪里**：`app.py` 每次生成音乐时由 `_log_session` 把 `ts / label / sys_emotion / sys_fatigue / sys_need / sys_music_spec` 追加进 **`results/session_log.jsonl`**（`sys_emotion`/`sys_fatigue` 来自 ④ 融合，`sys_need`/`sys_music_spec` 来自 ⑤ `LLM.infer`）。分析时按 **`label`（=参与者编号）** 从日志里筛出该被试那几条、填进 `user_study.csv`——**不用手抄 UI**。
> **区分「实验数据」与「平时测试数据」**：UI 顶部有「🧪 会话标签」输入框——**实验时填参与者编号（P01…），平时测试保持 `test`**，按 label 一筛就分开。日志位置可用环境变量 `STUDY_LOG=<path>` 覆盖。
> **隐私**：日志只存**标签/文本**，**不含**人脸图像或录音；`results/` 已 gitignore。

**可视化留存（⑦ GradCAM + ② 疲劳，随 app 自动生成、按 label 保存）**：这两张图**每次生成音乐时自动产出**（GradCAM 由 `face_emotion.gradcam` 一直在跑并显示；疲劳可视化由新增的 `fatigue.visualize` 现算），并保存到 **`results/session_media/<label>/gradcam_<ts>.png` 与 `fatigue_<ts>.png`**。

> ⚠️ **默认关闭 + 隐私须知**：这两张都是**人脸衍生图（含被试面容）**，与"不保存人脸"相冲突，故**默认不保存**——只有显式开启且**已在同意书里勾选同意**时才留存（附录 A 已含此条）。原始摄像头帧/录音**始终不存**，只存这两张分析用的标注图。归 **M4**（app.py）+ **M1**（GradCAM 语义）/ **M4**（疲劳）。

**如何开启 / 关闭可视化保存**：靠环境变量 `STUDY_SAVE_MEDIA` 控制（启动 `app.py` 前设）：

```bash
STUDY_SAVE_MEDIA=1 python app.py          # 开启（Git Bash / Linux / macOS）
$env:STUDY_SAVE_MEDIA=1; python app.py    # 开启（Windows PowerShell）
set STUDY_SAVE_MEDIA=1 && python app.py   # 开启（Windows cmd）
python app.py                             # 关闭（默认；不设该变量即不保存）
# 改保存位置：再设 STUDY_MEDIA_DIR=<某目录>（默认 results/session_media）
```

## 4. 问卷（5 点 Likert，逐题）

沿用最初规划的 Q1–Q5，补一题可选的疲劳 Qf。锚点固定、正向计分。

- **Q1 情绪感知准确性**：「系统感知到的情绪，和你当时**实际感受**一致吗？」 1=完全不一致 … 5=完全一致
- **Q2 音乐生成质量**：「生成的音乐**符合你当时的心情**吗？」 1=完全不符 … 5=非常符合
- **Q3 推理价值**（二选一 + 开放）：「刚才两段音乐——一段来自完整推理、一段来自简单查表——哪段让你觉得**更贴心**？」☐前 ☐后 ☐差不多；「为什么？」
- **Q4 完整系统有效性**：「用完这一轮后，你的情绪状态有变化吗？」 1=更糟 … 3=没变化 … 5=明显变好
- **Q5 隐私接受度**：「被摄像头和麦克风感知，你觉得侵扰感强吗？」 1=侵扰感很强 … 5=完全不介意
- **Qf 疲劳贴合（可选）**：「这段音乐贴合你此刻的**精力/疲惫程度**吗？」 1=完全不贴合 … 5=非常贴合
- **开放**：最喜欢的一点 / 最想改进的一点 / 对"让它一直读你摄像头麦克风"的想法。

## 5. 结果如何分析

**N≤10 → 描述性为主 + 趋势 + 定性引语**，不追统计显著。

| 假设 | 用哪些数据 | 怎么分析 | 判定 |
|------|-----------|----------|------|
| **H1** | Q1_perc_match | 中位数/均值 + 打 4–5 的比例；列几个 `sys_emotion` vs 自评的具体例子 | 中位数 ≥4 且 ≥60% 打 4–5 → 支持 |
| **H2** | Q2_music_fit | 同上 | 中位数 ≥4 → 支持 |
| **H3** | Q3_pref + Q3_reason | 统计 tom / std / tie 计数；对 reason 做**主题编码** | tom ≥60% → 支持；引原话佐证 |
| **H4** | Q4_mood_change | 分布（更糟/没变/变好的人数）+ 中位数 | 中位数 ≥3.5、正向为主 → 支持 |
| **H5** | Q5_privacy | 分布 + 中位数 | 描述呈现，接 critical reflection（隐私代价） |
| **H6** | Qf_fatigue_fit（尤其疲惫子集） | 中位数 + 分组看 | ≥3.5 → 疲劳感知有价值 |

**分析步骤**：
1. 汇总 `user_study.csv`，每题算**中位数 + 均值 + 各分值人数分布**（画简单条形图）。
2. Q3 做偏好计数（可选符号检验 vs 50%，仅作参考）。
3. **定性**：把 Q3_reason 与访谈的 like/improve 做主题编码（两人独立、汇总主题），用真实引语支撑数值。
4. **可追溯例子**：挑 2–3 个 `sys_emotion → sys_need → sys_music_spec → 用户打分` 的完整链，放进报告/slides（很有说服力）。
5. 报告每条假设给"支持/趋势/不支持" + 图 + 引语，并显式写样本量与局限（§7）。

## 6. 情绪如何控制（沿用最初规划）

强迫别人"进入某情绪"很难，用两方案组合：

- **方案 A（主观匹配度，最简单）**：不控制情绪，让用户带着当下真实状态正常用，结束问 Q1"感知与实际是否一致"。**这是匹配度评分，不需事先控制。**
- **方案 B（情绪诱导，有学术依据）**：用前播一段特定情绪短片（搞笑→开心、压抑→悲伤、紧张→焦虑），立刻测试。这是 **Emotion Induction**，心理学标准做法，报告需注明用了哪种 induction protocol。

**每人只做一种方案**（同时做两种会拉长时长、且两种协议测的东西不同、不好比），**在整个样本层面做 A+B 结合**：

- **默认用 B（诱导）**，并**提前给 ~10 人分配轮换的目标情绪**（如 2 sad / 2 anxious / 2 angry / 2 happy / 2 tired）——保证覆盖主要情绪、真正测到系统对不同情绪的推荐路径。只有 ~10 人时，自然到场的心情多半平淡/中性，测不到 sad/angry/fear 这些关键路径。
- **对到场时已明显处于某种强烈真实情绪的人，用 A（自然）**：此时诱导多余、反而不真实，直接让 TA 带真实状态用、问 Q1 匹配度即可。
- 分配表由 **M4 提前定**、当天可微调；每人的方案 + 目标情绪记进 `induction` 列。

把"情绪 ground truth 的主观性"诚实列为 limitation（§7）——正好是 critical reflection 的加分点。

## 7. 局限性（写进报告）

- **情绪 ground truth 主观**：用自我报告评"感知准确率"，而人的自我情绪报告本就不完全可靠——这是我们选主观匹配度问卷（而非硬 ground truth）的原因，诚实列为方法论 limitation。
- **样本小、便利抽样**（≤10，同学为主）→ 结论是趋势性、不可泛化。
- **新奇效应**：初次体验的新鲜感可能抬高评分；单次使用不反映长期陪伴价值。
- **MusicGen 延迟**：musicgen-small 在本机 GPU 上实测**每段 8s clip ≈ 20s**（进程首次还有 ~30s 模型加载）。**每场实验开始前先随便跑一次「预热」**（吃掉一次性加载），之后每次生成就只等 ~20s；每人 2 次生成（步骤 2 + Q3）≈ 40s 等待——嫌慢可把 `music_gen.DEFAULT_DURATION_SEC` 从 8 降到 5（≈13s/段）。这段等待可能影响体验评分，报告里如实说明。
- **主试非盲**：靠统一脚本降低引导；被试对 Q3 的两段来源保持不知情。

## 附录 B · 主试脚本（逐句念，保证一致）

- **开场**：「谢谢参加。你会实际用一下这个音乐陪伴系统：对着摄像头说一句你现在真实的想法，它会判断你的情绪、生成一段音乐放给你听。全程我们**不会保存你的脸或声音**，只记录它判断的情绪和你的打分。随时可以停。」
- **（方案 B 诱导）**：「先看一小段视频，帮你进入一种状态。」（播 60–90s）
- **使用（手动模式）**：「现在对着摄像头拍一张、再录一句你此刻真实的想法，什么都行。我来点『运行』。」（主试点 ▶ Run Pipeline once，播放音乐、让 TA 听完）
- **反馈**：「它感知到的情绪和你实际感受一致吗（1–5）？这段音乐符合你的心情吗（1–5）？用完这一轮，你的情绪有变化吗（1–5）？」
- **Q3 对比**（主试把「⑤ Reasoning mode」切到 Standard、同一输入再点一次运行）：「我再给你听一段——这段是用**简单查表**方式为你刚才同样的状态做的。两段哪段更贴心？为什么？」
- **收尾**：「这段音乐贴合你此刻的精力/累的程度吗（1–5）？被摄像头麦克风一直感知，你觉得侵扰感强吗（1 很强—5 完全不介意）？最后：最喜欢哪点？最想改进什么？」
